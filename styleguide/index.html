1. General Concepts

Build HTML from Pre-made UI Pieces

New pages should not generally require much additional CSS. We should spend most of our time caring for our reusable framework components so new pages can quickly be built from them. More systems thinking. Less page-level building.

References:

  • http://www.slideshare.net/stubbornella/object-oriented-css

Separate Structure and Skin

Many components have a similar layout but different visual treatments. By separating the structural properties we can reuse them.

References:

  • http://www.slideshare.net/stubbornella/object-oriented-css

Separate Container and Content

As much as possible, we make pieces of UI that are unopinionated about:

  • the type of elements they can contain
  • the container they are located in

They will behave predictably regardless of how they are nested in the HTML. This means we end up with elements that can be safely combined in many different ways.

There are occasional exceptions to the rule but they are always justified.

References:

  • http://www.slideshare.net/stubbornella/object-oriented-css

Don't Overdesign, Don't Over-engineer

We make the necessary tradeoffs to keep our framework simple. We strike a balance between the features we need and the costs of implementation, maintenance and scalability.

Example one:
We don't write overly complex Sass mixins for the sake of keeping our code a tiny bit DRYer. Every time a co-worker has to read through our mixins, it costs them time and energy. Often more than the actual code savings are actually worth. Don't write a five line mixin to generate 10 lines of CSS. In that case copypasting is faster and easier to read.

Example two:
We don't make responsive variations of the same component look drastically different because it will force us to make that component too flexible and therefore, too complex. The implementation and maintenance costs are greater than the value for the user.

References:

  • http://www.sitepoint.com/keep-sass-simple/

CSS and HTML that is Useful for Developers

Nicholas Gallagher outlined this in the following post:

http://nicolasgallagher.com/about-html-semantics-front-end-architecture/

Regarding class names:

not all semantics need to be content-derived

Regarding the fear of overusing so-called "unsemantic" HTML (sometimes known as "divitis"):

We shouldn’t be afraid to include additional HTML elements if they help create more robust, flexible, and reusable components. Doing so does not make the HTML “unsemantic”, it just means that you use elements beyond the bare minimum needed to markup the content.

Further references:

  • http://cssguidelin.es/#naming

Clarity over Brevity

We prefer to write class names that are as informative as possible about what they do and what they are related to (even if this means making them more verbose).

We rarely use cryptic abbreviations.

Mobile First

By default we design for the smallest screens and breakpoints allow us to override default values on larger screens. Therefore almost all our breakpoints are min-width.

Sizing Units

We establish font-sizes in REMs so we can change them easily on mobile.

Margins, paddings, etc. use px instead of REMs because we want them to be independent from font sizes. For example, on mobile we may want smaller text but more padding (to improve touch target size).

Flexbox Grid

Our layout object uses flexbox. It allows us to do everything traditional grids can and much more.

  • https://philipwalton.github.io/solved-by-flexbox/

Browser Support

We support the 2 latest versions of IE. Most of what we've built works in "modern browsers" several versions back. We do not support minority browsers like Opera.

Minimal Nesting

Whenever possible we avoid nesting in order to prevent unnecessary specificity.

!important

!important is used -almost- exclusively in utility classes (named "trumps" in ITCSS).

Putting it anywhere else would make it difficult to override. Utilities are the last layer in ITCSS precisely because they are meant to override anything else. The risk of them needing to be overridden is practically nonexistent.

2. Methodologies

ITCSS

We follow the ITCSS methodology. This involves organising styles according to specificity. For example, we don't put styles for form elements and form classes in the same file. Element styles are located in one file and classes in another. Since element selectors are less specific, they are imported earlier than classes.

Layers described in the ITCSS methodology:

  1. Settings
    Variables and config that are used globally throughout the project.
  2. Tools
    Globally-available functions and mixins.
  3. Generic
    Far-reaching, very low specificity selectors. It is usually used for reset, normalisation, etc.
  4. Elements
    Bare, unclassed HTML elements. What does an H1 element look like without a class on it? What does an A element look like without a class on it?
  5. Objects
    Non-cosmetic design patterns. Any piece of UI that is purely structural, and doesn’t have any look-and-feel.
  6. Components
    Discrete, self-contained pieces of UI. Your dropdown menu, your carousel, your price-comparison module.
  7. Trumps
    Utilities, helpers, overrides, and hacks. It takes its name from the English word trump, which means to beat (someone or something) by saying or doing something better.

Objects

Objects are structural abstractions that mainly handle layout. Example: our layout object is a grid system completely devoid of any skin. Its sole purpose is to distribute elements throughout the page.

Objects usually have no "look and feel" properties. If they ever require any visual styling it is only because that styling is unquestionably essential to their purpose. And even then, any colors, borders, etc. are kept neutral. They should look like they could be used on any other site. A table, for example, will require some kind of styling to help users clearly scan each row.

Components

Components are where most of the "look and feel" is located. They are the specific pieces of UI that non-developers would normally refer by name to when speaking about an interface: "main menu", "CTA button", "testimonial", etc.

Components will sometimes depend on objects but they can also exist on their own.

If you are building an object and it seems to have some opinionated "look and feel" properties that might be an indication it should be a component. If you are in doubt about whether something should be an object or a component, then use a component just to be on the safe side.

States and Modifiers

Modifiers are permanent variations of a component or an object. Examples could be .c-button--large, .c-button--small, .c-button--primary, .c-button--secondary.

States are temporary variations of a component or an object, that are triggered by JavaScript. Examples could be .is-open, .is-closed, .is-collapsed, .is-expanded.

Utilities

Utilities are high-specificity classes meant for occasional overrides. They tend to override a single CSS property by making use of !important. You should rarely need them. Watch out if you find yourself:

  • adding several to the same element
  • adding the same ones to similar elements in different contexts

In these cases you should stop and consider whether the styling you need should simply be part of a new modifier class for the object or component you are tweaking.

BEM-like Class Names

BEM-like naming conventions help us identify at a glance:

  • The root element of an object or component.
  • Child elements that are context-dependent: they only make sense in the context of the parent they belong to.
  • Optional modifiers that alter appearance or behavior.

Basic naming conventions:

.prefix-component {}
	.prefix-component--modifier {}
	.prefix-component__child {}
	.prefix-component__child--modifier {}

Using our layout object as an example:

.o-layout {}
	.o-layout--vertical {}
	.o-layout__item {}
	.o-layout__item--shrink {}

We add extra prefixes to our classes to help us further understand the function they perform:

  • o- denotes an object
  • c- denotes a component
  • u- denotes a utility class
  • is- , has- denote temporary behaviors often invoked by JS (do not confuse with modifier classes)
  • js- denotes a class that is used exclusively as a hook for Javascript
  • qa- denotes a class used for quality assurance
  • t- denotes a class that applies a specific theme/skin

Another advantage is good autocompletion in our code editor and improved searching.

Here are some component examples.

Parent and child elements:

.c-nav {} // Parent
	.c-nav__item {} // Add to any LI element
	.c-nav__link {} // Add to any A element

Modifiers for a parent element:

.c-box // Parent
	.c-box--primary {} // Primary brand colour
	.c-box--secondary {} // Secondary brand colour

Modifier for a child element:


	.c-box__link--primary // Change colour of a link in a .c-box--primary

References:

  • http://cssguidelin.es/#naming-conventions
  • https://en.bem.info/method/definitions/

3. Variable Naming Conventions

<layer> refers to the ITCSS layer this variable belongs to. <file> refers to the name of the file the variable should be located in.

Please see examples below for further explanation.

Variables for Enabling Modifiers

Pattern:


	$bc-<layer>-<file>-enable-<feature|modifier>

Examples:


	$bc-o-layout-enable-gutter
	$bc-o-layout-enable-gutter-tiny
	$bc-o-layout-enable-gutter-small
	$bc-o-layout-enable-gutter-large
	$bc-o-layout-enable-gutter-huge

Variables for Properties (not colors)

Pattern:


	$bc-<layer>-<file>-<element>-<property>-<unit>-<state>-<modifier>

Examples:


	$bc-e-page-font-family
	$bc-e-page-line-height-unitless
	$bc-e-page-font-size-percentage

Variables for Colors

Pattern:


	$bc-<layer>-<file>-<element>-color-<background|border|text>-<state>-<modifier>

Examples:


	$bc-c-buttons-color-background
	$bc-c-buttons-color-background-focus
	$bc-c-buttons-color-border
	$bc-c-buttons-color-border-focus
	$bc-c-buttons-color-text

Other "Keyword" Variables

When a kind of "keyword" is used to designate the thing being modified.

Pattern:


	$bc-<layer>-<file>-<keyword>-<modifier>

Examples:


	$bc-o-layout-gutter
	$bc-o-layout-gutter-tiny
	$bc-c-buttons-spacing-unit
	$bc-c-buttons-spacing-unit-tiny

Variables for Use in Mixins

Pattern:


	$bc-<layer>-<file>-<keyword>

Examples:


	$bc-s-responsive-breakpoints: (
	t: "screen and (min-width: 30em)",
	s: "screen and (min-width: 48em)",
	m: "screen and (min-width: 64em)",
	l: "screen and (min-width: 80em)",
	h: "screen and (min-width: 90em)",
	g: "screen and (min-width: 120em)",
	sl: "screen and (max-device-width: 48em) and (orientation: landscape)"
	);


	$bc-u-spacing-margin-prefix
	$bc-u-spacing-padding-prefix
